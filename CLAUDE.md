# 문서 작성 가이드

모듈화된 제안서 작성을 위한 스타일 가이드

---

## 📁 프로젝트 구조

Projects path 내부에 프로젝트 이름 별 디렉토리 구성
프로젝트 내부 제출서류(Main_Documents)와 참고서류(Ref_Dcouments)로 구성
제출서류는 같은 폴더내 document_structure.md 의 형태에서 벗어나지 않아야함 (절대 준수))

```
Projects/
├── Project_name/
│   ├── document_structure.md 
│   ├── Main_Documents/ 
│   └── Ref_Documents/

```

## ✍️ 문장 스타일 규칙

### 필수 어미
```
✅ "~함", "~임", "~을 목표로 함"
✅ "~구축", "~확보", "~달성", "~실현"
❌ "~합니다", "~입니다", "~할 것입니다"

### 존댓말 어미 최소 교정 원칙
```
- "~합니다", "~입니다", "~될 것입니다" 등 존댓말 어미만 "~함/~임/~을 목표로 함"으로 교정
```

### 문장 패턴
```
[문제/배경] + [해결책] + [기대 결과]

예시:
"LLM이 가중치에 모든 정보를 저장하는 한계를 극복하기 위한(배경)
장기기억 모듈을 기반으로(해결책) 효율적인 정보 저장과 학습이
가능한 환경을 마련함(결과)."
```

### 기술 설명 패턴
- **선행연구**: "DeepSeek Engram, Gemma-3n의 ~을 정밀 비교"
- **수치화**: "20% 이상", "150B급", "64K 토큰"
- **비교**: "기존 대비 3배", "1/5 수준으로 억제"
- **방법론**: "~를 활용하되", "~로 오프로딩"

## ⚠️ **정부 제안서 필수 스타일 (절대 준수)**

### 음슴체 철저 사용
```
✅ 정부 제안서 스타일 (음슴체)
"메모리 모듈을 통합하여 성능을 향상시킴"
"K-AI 사업 목표 달성을 위한 평가 체계를 수립함"
"글로벌 모델 대비 20%p 우위를 확보함"

❌ 일반 문체 (절대 사용 금지)
"메모리 모듈을 통합하여 성능을 향상시킵니다"
"K-AI 사업 목표 달성을 위한 평가 체계를 수립합니다"
"글로벌 모델 대비 20%p 우위를 확보합니다"
```

### 이모지 사용 금지
```
❌ 절대 사용 금지
"## 🎯 최종 목표"
"✅ 달성 완료"
"🚨 주의 사항"
"📊 성능 비교"

✅ 정부 제안서 스타일
"## 최종 목표"
"달성 완료"
"주의 사항"
"성능 비교"
```

**예외:**
- CLAUDE.md 자체(가이드 문서)는 가독성을 위해 이모지 허용
- 실제 제안서 파일(sections/)에서는 **절대 금지**

### 표 작성 시 이모지 금지
```
❌ 잘못된 예시
| 모델 | 성능 | 특징 |
|------|------|------|
| 본 과제 | 82% | ✅ 최고 |

✅ 올바른 예시
| 모델 | 성능 | 특징 |
|------|------|------|
| 본 과제 | 82% | 최고 |
```

## ⚠️ 주의사항

### 하지 말아야 할 것 (정부 제안서)
```
❌ 서술형 어미 ("~합니다", "~입니다") → 음슴체 필수
❌ 이모지 사용 (절대 금지) → "목표", "달성", "주의" 등 텍스트로만
❌ 수동태 ("~되었다", "~진행될 예정이다")
❌ 모호한 표현 ("많은", "빠른", "좋은")
❌ TODO 남기기 (완성 후 제거)
❌ 4단계 이상 불렛 중첩
❌ 특정 기술에 과도하게 고착 ("반드시 ~만 사용")
❌ 벤치마크에서 비현실적 목표 (Kimi K2.5, Gemini 3 Pro 수준)
```

### 반드시 해야 할 것 (정부 제안서)
```
✅ 음슴체 ("~함", "~임") 철저 사용
✅ 이모지 제거 (정부 제안서 표준)
✅ 능동태 ("~구축함", "~달성함")
✅ 구체적 수치 ("20% 이상", "150B급")
✅ 섹션 간 연결성 명시
✅ 빌드 후 전체 검토
✅ 벤치마크 현실적 목표 
✅ 기술 명시 시 대안 가능성 열어두기 ("~등", "유사 기법")
```
